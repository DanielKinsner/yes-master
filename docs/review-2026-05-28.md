## Findings

### 1. `setExportLufsPreview` hardcodes `editingAlbumIntent: false` — live push skipped in album mode

**File:** `src/hooks/useTrackMaster.ts` ~line 1544  
**Status:** CONFIRMED

```typescript
shouldPushLiveChainForSettingsEdit({
  trackId: selectedTrackId,
  editingAlbumIntent: false,   // ← hardcoded
  loadedTrackId,
  loadedKindByTrack,
  overrideAlbum,
})
```

The other two call sites (`updateSettings`, `applyUserPreset`) compute `editingAlbumIntent` as `mode === "album" && !overrideAlbum.has(id)`. With the hardcode, when in album mode and a following track (not in `overrideAlbum`) is loaded as Mastered, the predicate falls through to the per-track branch, finds no matching entry for `selectedTrackId` (a different track is playing), and returns `false`. No `updateChain` call fires.

**Fix:** Replace the hardcode with the same expression used at the other call sites:
```typescript
editingAlbumIntent: mode === "album" && !overrideAlbum.has(selectedTrackId as TrackId),
```

No test covers this scenario.

### 2. `throw playbackErrorMessage(err, kind)` throws a string, not an Error

**File:** `src/hooks/useTrackMaster.ts` ~line 1298  
**Status:** CONFIRMED (latent)

```typescript
} catch (err) {
  throw playbackErrorMessage(err, kind);  // returns string, not Error
}
```

`togglePlay`'s outer catch does `setError(String(err))`. `String("a string")` is identity, so the message survives correctly — but only by accident. Any future handler that calls `err.message`, checks `err instanceof Error`, or passes through a logging / telemetry layer that distinguishes Error objects will silently get `undefined` or the wrong branch. The original stack trace is discarded.

**Fix:**
```typescript
throw new Error(playbackErrorMessage(err, kind));
```

### 3. `applyExplicitLoudnessTarget` skips effective-value copy when switching from named profile to Custom

**File:** `src/lib/settings-transitions.ts` ~line 125  
**Status:** PLAUSIBLE

The old center quick-select path called `applyDeliveryProfileSelection(prev, "custom")` first, which copies the current profile's effective `ceiling_dbtp`, `bit_depth`, and `target_sample_rate` into `advanced` before switching. `applyExplicitLoudnessTarget` only writes `delivery_profile: "custom"` and `lufs_offset_db`, leaving those fields at whatever `advanced` holds.

**Scenario:** User's last Custom session had `ceiling_dbtp: -3, target_sample_rate: 44100`. They switch to StreamingUniversal (owned: -1 dBTP, 48 kHz) but never manually re-enter the advanced panel. They then pick "Off / Natural" from the center quick-select. The export lands on Custom with stale ceiling and sample rate from the prior Custom session.

Needs verification against the full `applyDeliveryProfileSelection("custom")` body.
