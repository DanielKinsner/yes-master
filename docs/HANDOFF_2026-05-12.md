## Pitfalls discovered this session

1. **Cargo test fails when `npm run tauri dev` is running.** Windows holds `target/debug/album-mastering-studio.exe` locked; cargo can't relink. Symptoms: `error: failed to remove file ... (os error 5)`. Workarounds:
   - Ask Dan to close the dev app.
   - Use `cargo check --tests` for type-only verification (doesn't relink).
   - Run the previous build's test binary directly: `./target/debug/deps/contracts-<hash>.exe <test_name> --nocapture`. Note: only includes tests present at that build's time.

2. **`eprintln!` from `audio.rs` does NOT reach Dan's `cmd` terminal.** Tauri-dev's subprocess spawn drops the binary's stderr from the parent's pipe (or at least, Dan saw no `[audio]` lines despite repeated slider drags). Lesson: don't use `eprintln!` for diagnostics the user needs to see. Use in-app affordances instead:
   - The `liveUpdateStats` counter pattern in `useTrackMaster.ts` is the working model.
   - Tauri events back to the frontend (see `RenderProgress` in `engine.rs`) are another working channel.

3. **React 18 batched updates trap.** Do NOT rely on side-effect assignments inside `setState((prev) => { x = mutate(prev); return ... })` for synchronous reads after the setState call. Compute the value from closure-captured state first, then setState with the result. This was the proximate cause of the live-update bug (see `a7bd6b0` for the fix pattern).

4. **Add `#[serde(default)]` to every new field on `MasteringSettings` (and similar persisted structs).** Old sessions written before the field existed must still deserialize. See `input_gain_db` and `output_gain_db` in `types.rs` for the pattern.

5. **Cargo.toml line-ending churn (LF ↔ CRLF) on Windows.** Don't commit it. `git checkout -- src-tauri/Cargo.toml` before committing if `git status` shows it as modified after a cargo run.

6. **DevTools is not reliable.** Dan can't open F12 while actually working. Don't propose "open DevTools and report what you see" as a diagnostic — build in-app affordances instead (badges, counters, status pills).

7. **`AdvancedPanel` has placeholder fields.** Five of eight Advanced controls are not wired into the chain at all. They're labeled "(coming soon)" in the UI as of `4076596`. Specifically: `lufs_offset_db`, `width`, `warmth`, `presence_air`, `compression_density` are read in nowhere. Wired fields: `ceiling_dbtp` (used in `ChainCoeffs::from_settings`), `bit_depth` (used in `write_wav`), `target_sample_rate` (currently NOT wired into resample — also a TODO).

8. **The decode cache is a single-slot LRU.** If Dan plays Track A, then Track B, then back to Track A, A's cache was evicted by B and will re-decode. For Track Master single-file workflow this is fine; for album sessions it'll re-decode on every track switch. Multi-slot LRU is a deferred refinement.

9. **`mastering_render` uses 4096-frame chunks** for progress reporting (`51ea09b`). The chain's per-frame state (limiter lookahead, biquad memory) flows correctly between chunks because we call into the same `chain` instance. Don't refactor this loop to use `Vec::chunks_mut().for_each()` without preserving state continuity.

## Open work — prioritized

### P0 — next slice candidates (objective, no listening required)

#### 1. Live clipping / output peak indicator

**Why it matters:** Dan flagged that "presets clipping on already-mastered material" is a UX disaster. The export receipt warns AFTER the fact; users need a live signal during slider drags. Standard DAW peak meter is the gold standard but requires streaming peak data from the audio thread to the frontend. Minimum viable: a "CLIPPING" pill that turns red when the post-output-gain peak exceeds 0 dBFS within the last ~100 ms.

**Design (suggested):**
- In `audio.rs::MasteringSource`, track `max(abs(frame_main))` per channel each frame. Store in an `Arc<AtomicU32>` (f32-as-bits) shared with `AudioThreadState`.
- In the audio thread's snapshot loop (line ~668 of `audio.rs`), swap-and-read the atomic to get peak-since-last-tick; reset to 0 in the swap.
- Add `peak_dbfs: f32` to `PlaybackSnapshot` and `PlaybackTick`.
- In `useTrackMaster.ts`, store the peak in transport state. Render a `<ClippingIndicator>` in the StaleBar that flashes red when peak > -0.1 dBFS.
- Unit test: feed a known signal through MasteringSource with a chain that pushes above ceiling, verify the peak atomic reflects it.

**Acceptance:** Dan can tell at a glance whether his current slider settings produce clipping, without exporting.

#### 2. Wire `width` (Advanced)

**Why:** Width is the easiest of the five unwired Advanced controls — M/S processing is a clean stereo transformation. Removes one "(coming soon)" label.

**Design:**
- In `MasteringChain::process_frame_inplace`, after EQ but before saturation, decode frame to mid/side: `mid = (L+R)/2; side = (L-R)/2`. Scale side by `width` (0 = mono, 1 = neutral, 1.5 = wider). Re-encode: `L = mid + side; R = mid - side`. Only execute when `channels == 2` AND `width.is_some()`.
- Tests: synth stereo signal (L = sine, R = -sine), verify width=0 → output is mono (L == R), width=1 → output unchanged, width=1.5 → side amplified.
- Remove the "(coming soon)" label.

#### 3. Wire `compression_density` (Advanced)

**Why:** A real compressor is missing from the chain. Currently the only dynamics shaping is the brick-wall limiter. A program-dependent compressor before the limiter would let Dan dial in glue without slamming the limiter.

**Design:** Bigger than Width — needs a proper attack/release envelope follower + gain reduction curve. See `docs/research/` for compression-design references. Probably a separate slice on its own (~300-500 lines).

#### 4. Wire `lufs_offset_db` (Advanced)

**Why:** Currently only used to populate `recommended_universal` in analysis. Should also drive an output-gain auto-adjustment at render time so the final master targets the LUFS the user selected.

**Design:** Post-render, measure the rendered file's integrated LUFS. Compute gain delta to hit the target. Re-apply gain (one-pass scaling, no re-limiter — accept that we might miss target by 0.5 dB).

#### 5. Wire `warmth` and `presence_air` (Advanced)

**Why:** Both are essentially extra EQ shelves with saturation flavor. Lower-priority than the controls above.

## A note on operating philosophy

What worked this session:
- **Small commits per slice, pushed immediately.** Easier to revert, easier to review, smaller blast radius.
- **Tests-first when the bug is unclear.** The `mastering_source_applies_live_coeff_updates_via_channel` test (eaeddc4) proved the backend was fine and narrowed the bug to the frontend in seconds — Dan's manual debugging took several rounds.
- **In-app diagnostic affordances.** The "live: N/M" counter (a7bd6b0) is a working pattern for verification without DevTools.
- **Honesty in UI labels.** Marking unwired controls as "(coming soon)" is better than letting them silently do nothing.

What didn't work:
- **`eprintln!` for diagnostics.** Wasted a turn assuming Dan would see them; he didn't.
- **Assuming React 18 setState semantics matched React 17.** Side-effect assignments inside setState updaters are unreliable.
- **Asking Dan to open DevTools while working.** He literally can't.

Apply these lessons to future slices.
