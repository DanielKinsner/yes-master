## What to test first thing

### 1. The sweep test (the original complaint)

Load a Mastered track, hit play, then **aggressively sweep**:

- Intensity slider for 10+ seconds.
- Each EQ band (especially MID and HIGH).
- Output gain.
- Compressor threshold / density.

Do it twice — once with **Preview LUFS off**, once with **Preview LUFS on**. Both should now feel responsive with no stutter and the audio should track the slider position with ≤30 ms perceived latency.

If you still hear stutter under sweep, capture the counters mid-sweep (see "Diagnostic counters" below) — that will tell us which fix is misbehaving.

### 2. The LUFS catch-up question

Change the LUFS target (delivery profile → custom target → bump a few dB) while audio is playing with Preview LUFS on. You'll hear the audible level ramp up/down to match the new target over ~30 ms. That's expected — the BS.1770 measurement runs on a background thread and the source crossfades to the corrected gain when it arrives.

You said earlier this is "almost tolerable." If it crosses into "annoying," the math shortcut I mentioned can make target-only changes instant (no measurement at all, just `new_scalar = old_scalar × 10^((Δtarget)/20)`). Say the word and I'll implement it.

### 3. Track-switch poison check (codex's P1)

The pre-fix bug: play Mastered on Track A with Preview LUFS on (so the cache fills with Track A gains), switch to Original on Track B, switch back to Mastered on Track B, move a knob with Preview LUFS still on. Pre-fix the cached Track-A gain could land on Track-B audio. Post-fix the cache is cleared at the boundary.

Verify by ear: do that sequence with two tracks of clearly different loudness, and check that the level after the knob move sounds right for Track B, not Track A.

### 4. Knob color sanity

Open the EQ panel. The three Tone Shape knobs should now color-match the three highlighted (ringed) EQ nodes:

- LOW knob = cyan = 200 Hz node = cyan.
- MID knob = purple = 1.5 kHz node = purple.
- HIGH knob = blue = 6 kHz node = blue.

If any of those three pairs don't visually match, the change didn't land.

## Diagnostic counters

I left `get_diag_counters` wired in as temporary instrumentation. From the Tauri devtools console (Cmd/Ctrl + Shift + I in the dev build), run:

```js
await window.__TAURI__.core.invoke('get_diag_counters')
```

What healthy numbers look like during a sweep with Preview LUFS on:

- `update_chain_dispatched`: rises with knob motion (one per dispatched IPC, post rAF-coalesce).
- `lufs_workers_spawned`: rises but slowly — should equal "distinct settings the user lingered on long enough for a measurement to start," NOT every knob micro-movement. If you sweep through 200 micro-positions and this counter climbs by 200, Fix C is broken.
- `lufs_workers_queued`: rises during sweeps as later updates land on the pending slot.
- `lufs_workers_applied`: rises when the user actually settles on a setting and that measurement is current.
- `lufs_workers_cached_only`: rises when a worker returned but the user already moved on (free cache fill — good).
- `lufs_workers_rejected_stale_epoch`: should stay at 0 unless you swap tracks mid-measurement, then climb by exactly the number of in-flight workers that didn't finish before the swap.
- `mid_fade_promotions`: should rise during EQ / Intensity sweeps (proves Fix A is engaging). Pre-fix this would have stayed at 0.

When you're satisfied the realtime path is healthy, tell me and I'll rip the counters out in one revert (`git revert 53d8c74`).
